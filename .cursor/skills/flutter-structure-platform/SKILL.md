# Platform & build

## iOS

- **Podfile** — `platform :ios, '15.0'`; `post_install` sets minimum deployment targets and permission-handler flags.
- **`ios/Flutter/Debug.xcconfig` / `Release.xcconfig` / `Profile.xcconfig`** — After `#include "Generated.xcconfig"`, **overrides** `FLUTTER_APPLICATION_PATH`, `FLUTTER_TARGET`, `PACKAGE_CONFIG` using `$(SRCROOT)/..` so Xcode never passes a broken absolute path (e.g. trailing space from `flutter pub get`). Each file `#include?`s the matching `Pods-Runner.*.xcconfig` (debug / release / profile); the Runner target’s Profile build uses `Profile.xcconfig`, not Release.
- **`Generated.xcconfig`** — Regenerated by Flutter; can contain machine-specific paths. Do not rely on it alone for `FLUTTER_APPLICATION_PATH` when using Xcode.
- Podfile also patches `DT_TOOLCHAIN_DIR` to `TOOLCHAIN_DIR` during `post_install`, which is important for newer Xcode/CocoaPods combinations in this repo.
- After native or plugin changes: `cd ios && pod install`.

## Android

- **`android/app/build.gradle`** — `compileSdkVersion` / `targetSdkVersion` aligned with plugins (e.g. 34 for current file_picker).
- **`android/gradle.properties`** — Use enough heap for Gradle transforms (e.g. `-Xmx4096M`). Prefer `android.enableJetifier=false` when deps are AndroidX-only; Jetifier + Flutter engine JARs can hit **Java heap space**.
- Do **not** manually add `io.flutter:flutter_embedding_*` in `dependencies`; the Flutter Gradle plugin supplies them (duplicates trigger `checkDebugDuplicateClasses` / extra Jetifier work).
- Manifest permissions in `android/app/src/main/AndroidManifest.xml`.

## Common failures

- **ProcessException / flutter assemble / No such file or directory** — Usually bad `FLUTTER_APPLICATION_PATH` (trailing space). Confirm xcconfig overrides exist; run `flutter pub get` from the real project folder (no trailing space in path).

## Update this skill when

Podfile, xcconfig override strategy, or Android SDK levels change.
